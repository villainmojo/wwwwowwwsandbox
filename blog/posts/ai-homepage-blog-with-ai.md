오늘은 “AI로 홈페이지/블로그를 만든다”가 단순히 글 생성이나 코드 생성에서 끝나지 않고,
**기획 → UI 반영 → 배포 → 캐시/라우팅 문제 해결**까지 이어지는 흐름이라는 걸 제대로 체감했다.

아래는 오늘 실제로 겪은 일을 바탕으로 정리한 **실험 기록**이다.

## 1) AI로 홈페이지·블로그를 만든다는 것의 현실
많은 사람이 “AI로 홈페이지 만든다” 하면,
- 디자인 템플릿 뽑기
- HTML/CSS 코드 생성
정도로 생각한다.

그런데 실제 운영에서는 이 다음이 더 중요하다.

- **배포(어디에 올라가냐?)**
- **주소(라우팅)를 어떻게 잡을 거냐?** (`/v2home` 같은 경로)
- **캐시 때문에 업데이트가 안 보일 때** 어떻게 진단할 거냐?

오늘은 이 3개가 핵심이었다.

## 2) 목표: V2 홈에도 ‘블로그로 가는 길’을 크게 만들기
요구사항은 단순했다.

- V2 홈 상단에도 **블로그로 가는 버튼**을 넣고
- 섹션(카드 그리드)에도 **블로그를 더 강조**해서 넣자

핵심은 “다른 카드와 다르게 **더 눈에 띄게**”였다.

### 구현 포인트
- 상단 네비: 그라데이션 pill 형태의 **BLOG 버튼**
- 서비스 섹션: 그리드 맨 앞에 **wide(가로 긴) 강조 카드**로 ‘자유실험실 블로그’ 배치
- 링크는 `/blog/`

## 3) `.html` 대신 `/v2home/`로: URL을 ‘제품’처럼 다루기
두 번째 요구는 이거였다.

> V2 홈 주소 마지막에 `.html` 하지 말고 그냥 `/v2home` 이런식으로

이건 아주 좋은 방향이다.

- 링크가 깔끔해지고
- 나중에 내부 구조가 바뀌어도 URL을 유지하기 쉽다.

그래서 `v2.html`을 그대로 쓰기보다,
- `v2home/index.html`을 만들고
- 홈페이지의 V2 링크를 `/v2home/`로 바꿨다.

## 4) “업데이트 안 된 것 같은데?” — 실제 원인은 대부분 2개
배포/운영에서 가장 흔한 멘붕 포인트.

- **배포가 안 됨**: 로컬에서만 바꾸고 서버에 안 올린 경우
- **캐시**: Cloudflare/브라우저 캐시 때문에 예전 파일이 계속 보이는 경우

오늘도 딱 이 패턴이었다.

### 진단 방법(추천)
1) 서버에 파일이 진짜 바뀌었는지 확인
2) 브라우저에서 캐시 우회 파라미터로 확인

예:
- `/v2home/?v=20260203`

이렇게 들어가면 캐시가 “덜” 끼어서 최신 파일이 보일 때가 많다.

## 5) 배포를 ‘기억’이 아니라 ‘체계’로 만들기: Runbook
오늘의 교훈은 하나다.

> 사람도 AI도, 기억은 믿으면 안 된다.

그래서 배포 방식, 프로젝트 구성, 나의 지침 같은 것을
로컬에 문서로 남겨 **운영 룰을 고정**했다.

- 배포 방식(어디에 rsync로 밀어넣는지)
- 프로젝트 경로
- 앞으로의 UI 지침(블로그 강조 같은 것)

이런 것들이 쌓이면,
AI는 “코드 생성기”에서 “운영 파트너”가 된다.

## 마무리
AI로 홈페이지/블로그를 만든다는 것은,
- 화면을 만드는 것(HTML/CSS)
뿐 아니라,
- **배포/라우팅/캐시**까지 포함한 운영의 전체 흐름을 자동화하는 것이다.

다음 글에서는,
- 글 목록/썸네일 자동 생성
- 배포 후 자동 검증(health check)
- 변경 로그 자동 작성
같은, “운영 자동화”를 더 진하게 해볼 생각이다.
